# プレビュー連動チェックボックス実装の試行錯誤と解決の記録

本アプリの目玉機能の一つである「プレビュー画面からのチェックボックス操作」の実装において、直面した課題と解決までの全プロセスを記録する。

## 1. 目的
Obsidianのように、プレビュー画面（Reading Mode/Split Mode）に表示されたチェックボックスをクリックすると、エディタ側のソーステキスト（`- [ ]` / `- [x]`）がリアルタイムに書き換わる機能を実装する。

## 2. 直面した課題（エラーと失敗の歴史）

### 第一の壁：インデックス（番号）のズレ
- **試したこと:** テキスト全体から `[ ]` を正規表現で探し、上から 0, 1, 2... と番号を振ってクリック箇所を特定しようとした。
- **失敗の原因:** メモの中にタスク以外の `[ ]`（ただのカッコなど）が混じると、プログラム側とレンダラー（描画側）で「数え方」にズレが生じ、クリックした場所とは別のタスクが書き換わってしまう問題が発生。

### 第二の壁：Reactのレンダリング・タイミング
- **試したこと:** 描画中にカウンター変数を増やす方法。
- **失敗の原因:** React 18以降の「Strict Mode」や再描画の仕組みにより、一回の表示でカウンターが複数回回ってしまい、インデックスが 3, 4, 5... と増え続けてしまう。結果として「2個目のチェックボックスが反応する」といった予測不能な挙動になった。

### 第三の壁：ライブラリの仕様変更と制約
- **試したこと:** `rehypeSanitize` の使用や、廃止された `sourcePos` オプションの導入。
- **失敗の原因:** 
    - `rehypeSanitize` がセキュリティのために `input` タグのイベント（onChange）を消し去ってしまい、無反応になった。
    - `sourcePos` は最新の `react-markdown` で廃止されており、Runtime Assertion エラー（強制終了）が発生。

## 3. 解決策：Source Mapping（行番号特定方式）

最終的に、**「描画プログラムが内部で持っている『行番号情報』を活用する」** という、もっとも堅牢な方法に辿り着いた。

### 解決のポイント
1.  **`li` コンポーネントのカスタマイズ:** 
    チェックボックスそのものではなく、それを包んでいる `li`（リスト項目）が持つ `node.position.start.line` データを取得。これが「このタスクはエディタの◯行目にある」という絶対的な住所になる。
2.  **`React.cloneElement` による動的な上書き:**
    レンダラーが作った静的なチェックボックスに対し、`onChange` イベントを後付けで注入し、取得した「行番号」を渡す仕組みを構築。
3.  **行ベースのテキスト置換:**
    インデックス（何番目）ではなく「◯行目の文字を置換する」というロジックにしたため、文章の構造や再描画のタイミングに一切左右されない完璧な連動を実現した。

## 4. 面接でのアピールポイント
- **問題解決能力:** 「ただ動かない」で終わらせず、ログを詳細に出力して「描画側とデータ側の数のズレ」を論理的に特定した。
- **技術への探究心:** ライブラリのバージョンによる仕様変更（sourcePosの廃止など）を調査し、常に最新のベストプラクティスを追求した。
- **UXへのこだわり:** 「チェックリストが操作できないのはUIとして欠陥である」という強い信念を持って、高度な実装をやり遂げた。
